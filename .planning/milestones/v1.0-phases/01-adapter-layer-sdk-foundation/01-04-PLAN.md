From packages/pi-coding-agent/src/core/backends/copilot-backend.ts:
```typescript
export class CopilotSessionBackend implements SessionBackend {
  readonly name = "copilot";
  constructor(private readonly clientManager: CopilotClientManager) {}
  async initialize(): Promise<void>;
  async createSession(config: BackendConfig): Promise<BackendSessionHandle>;
  async resumeSession(sessionId: string, config: BackendConfig): Promise<BackendSessionHandle>;
  async shutdown(): Promise<void>;
}
```

CopilotSessionBackend.createSession returns a CopilotSessionHandle (implements BackendSessionHandle) that wraps the SDK session with send/subscribe/destroy/abort methods. The tool bridge (bridgeAllTools) is called inside createSession, so tools are automatically bridged.

CRITICAL DESIGN CONSTRAINT: createAgentSession currently returns CreateAgentSessionResult which contains `session: AgentSession`. The AgentSession class is deeply coupled to the Pi Agent runtime (it wraps an Agent instance, SessionManager, SettingsManager, RetryHandler, etc.). For Phase 1 gap closure, the copilot path CANNOT return a full AgentSession — it must return the BackendSessionHandle and enough metadata for callers to manage. The approach: create a thin CopilotAgentSession adapter class that wraps BackendSessionHandle and exposes the minimal AgentSession-compatible interface needed for workflow commands.

However, a simpler approach exists: since Phase 1's success criteria only require that the copilot path creates and routes a real session (not that all commands work end-to-end through it), we can wire the copilot backend to create and hold a real session, expose the BackendSessionHandle on the result object, and still create a Pi AgentSession for the primary `session` field (for backward compat). The copilot session handle is available as an additional property for early adopters/testing.

CHOSEN APPROACH (minimal gap closure): Wire copilotBackend.createSession() to actually execute (not void), store the returned BackendSessionHandle on CreateAgentSessionResult as an optional `copilotSessionHandle` field, and keep the Pi AgentSession in the `session` field for backward compatibility. This satisfies SC-1 (switching creates a real copilot session), SC-2 (session is usable via handle), SC-3 (resume path also wired), and SC-4 (tools bridged inside createSession). Full AgentSession replacement on the copilot path is a Phase 3 concern.
